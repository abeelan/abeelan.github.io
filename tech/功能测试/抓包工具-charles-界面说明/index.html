分为四个区域（按图示分线）：

  1. 视图展示区域
  2. 请求/响应 展示区域

视图展示区域

推荐使用结构化视图，测试过程中只看自己相关业务的 host 请求即可

还可以在 Charles - Preferences - user Interface 内设置横向 或 纵向布局。

推荐纵向，即上图侧边栏的方式展示。

请求/响应 展示区域

  • overview：概述，请求的基本信息展示
  • request：请求详情
  • response：响应详情
  • summary：总览，选中多请求后会有汇总信息展示
  • chart：请求耗时图形化展示
    • 请求-发送（上传）请求所花费的时间（深蓝色）
    • 延迟-等待网络延迟或服务器上的处理时间所花费的时间（蓝色）
    • 响应-接收（下载）响应所花费的时间（浅蓝色）
  • notes：来源信息、错误信息、映射信息、Rewrite 等

求/响应 展示类型设置

  • View Request As
  • View Response As

有很多可选类型，比如：HTML、CSS、JSON等，根据具体的业务返回数据类型来选择。

其他设置

Focused Hosts

描述：「聚焦」功能，将自己关注的域名与其他域名分开展示。

使用：点击该选项，弹出的会话框内，点击 Add，填入常用的/关注的域名，点击确定。

示例：右图，设置的域名会单独展示出来，其他域名会收录到 Other Hosts 内

Viewer Mappings

描述：「查看器映射」，将指定域名下的请求返回值，映射为固定格式 xml（举例，json 也行）。

使用：右图，点击该选项，弹出的会话框内，点击 Add，填入需要映射的域名，点击确定。

该域名下所有请求返回的响应值，最终都会以 XML 格式展示。

Protobuf Settings

ProtoBuf （Google Protocol Buffer）是由 google 公司用于数据交换的序列结构化数据格式，具有跨平台、跨语言、可扩展特性，类型于常用的 XML 及 JSON，但具有更小的传输体积、更高的编码、解码能力，特别适合于数据存储、网络数据传输等对存储体积、实时性要求高的领域。以 .proto为后缀，有自己的编译器 protoc, protoc2 和 protoc3 版本.

Recording Settings

点弹出设置弹框，包含：

  • Options
  • Include
  • Exclude

Tab - 1 ：Options

1️⃣ Recording size limit（MB）

这个选项是限制请求及响应大小，如果超出 100 MB，则过滤掉，不会录制在 Charles 工具内；

100 这个数字可以随意设置

2️⃣ Limit recording history

这个选项是设置同一请求最大展示数量，如果开启选项，同一请求再次请求，会自动删除第一次请求的数据，所以为了避免数据丢失，建议不开启此功能或数字设置大一点

3️⃣ Limit WebSocket transaction history

这个选项类似 2⃣️，是针对 WebSocket 的限制

Tab - 2 ：Include

仅录制当前列表内添加的域名请求，如果不填或全部取消选中，则将所有的请求录制到 Charles 内

添加方法：

点击 Add → 弹出 Edit Location → 域名粘贴在 host 内，点击任意区域自动填充 → 点击确定

Tab - 3 ：Exclude

与 Include 相反，仅录制当前列表外的其他域名请求

如果不填或全部取消选中，则不排除任何域名的请求

添加方法与 Include 一致

Proxy Settings

Tab - Proxies

HTTP Proxy：

  • Port：设置端口号，默认 8888，手机连接代理时填写的端口号需要与此处保持一致，如果设置了固定端口，其他服务就不能再使用这个端口号，否则冲突；
  • Use a dynamic port：是否设置动态端口，如果勾选则监听动态端口，可以避免与计算机上运行的其他网络服务冲突，如果开的端口很多且不清楚端口号，可以勾选使用这个功能；
  • Enable transparent HTTP proxying：是否启动透明代理，可以先百度了解下什么是透明代理，在Charles 上具体有什么用处，我没搞懂。

SOCKS Proxy：

Socks 服务器通常是作为一个代理服务器，充当一个中间代理的角色，与客户端以及真正的服务端建立 TCP 连接，然后负责对客户端与服务端之间的流量进行路由和转发。一个 socks 代理服务器能够与任何一种网络协议进行工作

No Caching Settings

No Caching 工具可防止客户端应用程序（如 Web 浏览器）缓存任何资源。因此，始终向远程网站发出请求，您始终可以看到最新版本。

**适用范围：**该工具可以作用于每个请求(选中 Enable No Caching 即可)，也可以仅对你配置的请求启用(启用 No Caching 的同时，请选中 Only for selected locations)。当用于选定的请求时，可以使用简单但功能强大的模式匹配将工具的效果限制为指定的主机和路径。

**工作原理：**No Caching 工具通过操纵控制响应缓存的 HTTP 请求头来防止缓存。从请求中删除 If-Modified-Since 和 If-None-Match 请求头，添加 Pragma：no-cache 和 Cache-control：no-cache。从响应中删除 Expires，Last-Modified 和 ETag 请求头，添加 Expires：0 和 Cache-Control：no-cache。

Block Cookies Settings

Block Cookies 工具阻止了 Cookie 的发送和接收。它可用于测试网站，就像在浏览器中禁用了 Cookie 一样。 请注意，网络爬虫（例如 Google）通常不支持 Cookie，因此该工具还可用于模拟网络爬虫网站的视图。

**适用范围：**该工具可以作用于每个请求(选中 Enable Block Cookies 即可)，也可以仅对你配置的请求启用(启用 Block Cookies 的同时，请选中 Only for selected locations)。当用于选定的请求时，可以使用简单但功能强大的模式匹配将工具的效果限制为指定的主机和路径。

**工作原理：**Block Cookies 工具通过操纵控制响应 Cookies 的 HTTP 请求头来禁用 Cookies。从请求中移除 Cookie 请求头，防止 Cookie 值从客户端应用程序（例如 Web 浏览器）发送到远程服务器。从响应中删除 Set-Cookie 请求头，防止请求设置客户端应用程序从远程服务器接收的 Cookie。

Map Remote Settings

**Map Remote：**将请求的主机或者域名重定向到配置的主机或域名。

  • **HTTPS：**Map Remote 工具可以将 HTTP 请求映射到 HTTPS 目标，反之亦然。
  • **站点匹配：**每个站点匹配可能包含协议、主机、端口和路径模式，以匹配特定的 URL，站点可能包括通配符。

Map Local Settings

**Map Local：映射本地文件，**将本地文件的内容返回给客户端。

比如要修改服务的某个数值或者字段，正常情况下需要操作数据库或者后端服务去修改

通过 map local 可以直接修改本地文件，映射到客户端。

**动态文件：**动态文件（例如包含服务器端脚本的文件）不会由 Map Local 执行，因此如果文件中有任何脚本，脚本将按原样返回到浏览器，这可能不是预期的结果。如果您想使用动态文件，就好像它们是远程网站的一部分一样，请参阅 Map Remote 工具。

**工作原理：**当请求与 Map Local 映射匹配时，它会检查与路径匹配的本地文件。它不包括查询字符串（如果有）。如果在本地找到所请求的文件，则将其作为响应返回，就好像它是从远程站点加载的一样，因此它对客户端是透明的。如果在本地找不到所请求的文件，那么该请求会像平常一样由网站提供，返回由真正的服务器提供的数据。

**站点匹配：**每个站点匹配可能包含协议、主机、端口和路径模式，以匹配特定的 URL。站点可能包括通配符。当您向此工具添加新站点时，可能会找到有关创建站点匹配的更多帮助。
